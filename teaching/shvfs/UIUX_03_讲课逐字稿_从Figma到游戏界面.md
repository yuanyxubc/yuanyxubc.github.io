# 讲稿：从 Figma 到游戏界面 —— 行业协作流程
配套文件：`UIUX_03_知识补充_从Figma到游戏界面.html`（共 28 页，讲稿按页码一一对应）

---

### Slide 1｜标题
大家好，今天想跟大家聊一个可能大家在校内做项目时不太会碰到，但一进工作室就会天天遇到的事：一个游戏界面，从最初的想法，到玩家真正能在游戏里点到它，中间到底经过了哪些人、哪些工具。我会分八部分讲：Figma 和 Unreal 到底是什么关系；Figma 阶段该做什么、避免什么；这个流程里都有谁参与；团队规模怎么影响这个分工；UI/UX 岗位面试一般怎么考；技术美术需要学什么、又是怎么被面试的；笔试题和校招社招的区别；最后聊聊申请学校的时候，作品集又是另一套什么样的标准。

### Slide 2｜Agenda
简单过一下今天的八个部分。第一，讲清楚 Figma 和游戏引擎（这里用 Unreal 的 UMG 举例）到底是什么关系。第二，讲 Figma 阶段具体该怎么做，才能让后面对接 UMG 更顺畅，也讲讲常见的坑要怎么避免。第三，把从设计稿到最终界面之间会经手的八个角色一个个拆开讲。第四，讲团队规模——五人的小团队和三百人的 3A 团队，分工方式完全不一样，我会举例说明。第五部分是找工作相关的，UI/UX 设计师岗位面试一般会怎么考。第六部分讲技术美术这个角色，学什么、又怎么被面试。第七部分聊聊笔试题一般考什么，以及校招和社招——也就是国外常说的 Junior 和 Senior Artist——到底有什么区别。第八部分，也是给还在学校的同学最直接相关的，聊聊申请欧美英国的设计院校时，作品集要求是什么，以及游戏设计专业和游戏交互专业在申请标准上的差别。

### Slide 3｜全局流程总览
先给大家一个全局的图。一个界面的诞生大概是这样一条线：先是 Game Designer 提需求，告诉大家这个界面要解决什么问题；然后 UI/UX Designer 在 Figma 里把它设计出来；UI Artist 把设计稿做成最终质量的美术资源；Technical Artist 把这些东西在技术上"翻译"成引擎能用的形式；UI Programmer 在引擎里把它实现出来、能点了；最后 Gameplay Programmer 把真实的游戏数据接上去，界面才算真正活起来。除此之外，本地化、QA、制作人是全程都在背后做支持和协调的，这个后面会讲到。

### Slide 4｜Figma 和 Unreal 是什么关系
这里说一下 Figma 和 Unreal 到底是什么关系。简单说，Figma 是拿来做设计和沟通的，产出的是视觉稿、交互流程图、设计规范，这些东西本身不是游戏里能跑的东西，是给团队看、给大家讨论用的。而 Unreal 的 UMG，才是真正运行的界面所在的地方，要在这里拖控件、拉锚点、写蓝图、接数据，这一步的产出才是玩家最终会看到、会点击的东西。所以简单理解就是，Figma 负责"想清楚"，UMG 负责"做出来能跑"。

### Slide 5｜为什么不直接在 UMG 里做
那可能有同学会问，为什么不干脆省掉 Figma 这一步，直接在 UMG 里画？这里列了几个原因。第一是迭代速度，Figma 改个颜色、挪个间距是几秒钟的事，UMG 里往往要重新编译、等热重载，几十版下来时间成本差很多。第二是评审沟通，策划、制作人不可能人人都装引擎，但发个 Figma 链接谁都能看能评论。第三，早期做多方案对比的时候根本不需要"真实"，Figma 上几分钟能出好几版给大家挑。第四是技能门槛，纯做视觉的设计师不需要先学蓝图这些引擎知识就能参与。最后是组件系统和分工协作，Figma 的组件库管理设计规范很成熟，而且把设计和实现拆开，才能让不同角色并行工作，不互相卡着。

### Slide 6｜从 Figma 到 UMG 发生了什么
那具体这个转换过程是怎么发生的呢，大概是四步。第一步，在 Figma 里出设计稿和规范，这一步产出的是"图纸"。第二步，把图标、九宫格边框这些视觉元素导出成贴图，文字颜色间距这些整理成规范文档。第三步，是最费工夫的一步，在 UMG 里重新搭建——注意这不是"导入"，是照着图纸重新盖一遍房子，因为容器结构、锚点、自适应这些在 Figma 里是没有的概念，必须重新设计。第四步，动效和交互也要重做，Figma 里的动画效果只是给人看个意思，真正的动效要在 UMG 的动画面板里重新做关键帧，交互逻辑要靠蓝图连线实现。现在也有像 Figma2UMG 这样的插件，能帮忙自动生成一部分 Widget 结构，省一些体力活，但复杂的动效和数据绑定，目前还是得靠人工完成。

### Slide 7｜Figma 阶段：该做 / 该避免
既然接下来要交给 UMG 那边落地，Figma 阶段其实是有一套实操建议的，能不能遵循直接决定了后面对接顺不顺畅。该做的事情：用 Auto Layout 去搭模块，这样后面对应到 UMG 的 Horizontal Box、Vertical Box 逻辑是通的；给每个元素标注清楚"锚点意图"，比如这个东西该贴左上角还是居中还是贴右下角；九宫格素材要预留足够的安全边角；画布上叠加一层安全区参考线，核心元素别贴边放；统一用一个目标分辨率作为设计基准；图层和组件命名要规范，别用 Rectangle 23 这种默认命名。反过来该避免的：别用 UMG 还原不了的复杂特效，比如多层混合的阴影或者叠加模糊；别随意命名图层，不然接手的人转换成本会暴涨；别做完全不考虑分辨率变化的自由拖拽定位；九宫格素材如果没留边距，一拉伸就会变形；别在团队里混用好几套不同的基准分辨率；也别把 Figma 的 Smart Animate 效果当成最终动效标准去交付，那只是个示意。

### Slide 8｜如果想自己动手做 UMG，还要补充学什么
那如果同学们不满足于只做 Figma，想自己动手把界面在 UMG 里跑起来，还需要补充学哪些东西呢？第一是蓝图基础，至少要能看懂、也能自己搭一个简单的 Event Graph，做点击跳转这类基础交互。第二是 UMG 的容器逻辑，Canvas Panel、Horizontal/Vertical Box、Size Box、Scale Box 各自的区别和使用场景要搞清楚。第三是锚点系统的实操，理解 0 到 1 的坐标系统，会拖 Anchor，并且要在不同分辨率下实际测试适配效果。第四是素材接入的规范，比如 Draw As Box 的九宫格设置，图片 Brush 怎么导入替换。第五是动画面板的基础操作，关键帧动画怎么做，Fade、Scale、Move 这类常见动效怎么搭出来。最后是版本控制和一点性能意识，Perforce 或者 Git 的日常操作要会用，避免跟别人的资源冲突，也要知道控件层级不能嵌套太深，大概了解 Invalidation Box 是干什么用的。

### Slide 9｜角色总览
讲完了工具层面的关系，我们来看人。刚才提到的这条流程线上，其实一共有八个核心角色在参与：Game Designer 定义需求，UI/UX Designer 在 Figma 里做设计，UI Artist 做最终视觉资产，Technical Artist 做技术和美术之间的桥梁，UI Programmer 在 UMG 里实现，Gameplay Programmer 提供后端数据，Localization 做多语言适配，QA 和 Producer 做测试与协调。接下来我们一个一个展开讲。

### Slide 10｜Game Designer
第一个角色，Game Designer，也就是我们说的策划或者产品负责人。他们是这条流程的起点，负责定义这个界面到底要解决什么问题，比如"玩家要能一眼看懂当前任务进度"。他们会写功能需求文档，列清楚这个界面需要展示哪些信息、有哪些交互方式、还有哪些边界情况需要处理，比如信息为空的时候界面该长什么样。他们也会参与后面的评审，确认设计稿是不是真的满足了游戏系统的需要。他们交给下一个人（UI/UX Designer）的东西，就是这份需求文档。

### Slide 11｜UI/UX Designer
第二个角色，就是我们今天的主角之一，UI/UX Designer。他们拿到需求文档之后，要把它转化成具体的布局、交互流程和视觉方案，在 Figma 里做出线框图、可点击的原型，还有设计规范，比如间距、字体、色彩这些标准。他们也要组织设计评审，跟策划、制作人反复对齐意见。这一步产出的是 Figma 设计稿、交互流程图和设计规范文档，会交给下一步的 UI Artist 和 Technical Artist。

### Slide 12｜UI Artist
第三个角色，UI Artist，也就是专门做视觉美术的人。有些团队里这个角色和 UI/UX Designer 是同一个人，但大团队通常是分开的。他们的工作是把设计稿做成最终画面质量的美术资源，比如图标、按钮、边框、背景图；还要专门制作九宫格素材，预留好可以拉伸而不变形的区域；如果界面有动态效果，他们也会用 Rive、Lottie 或者 Spine 这类工具做出动态元素。他们最后交出的是一整包切好的贴图资源，给 Technical Artist 或者 UI Programmer 去导入引擎。

### Slide 13｜Technical Artist
第四个角色，Technical Artist，技术美术，这是很多同学不太熟悉、但其实非常关键的岗位，因为他们是设计和工程之间的桥梁。他们的工作包括把 Figma 稿子在引擎里翻译成真正的 UMG 结构，设置好容器、锚点、九宫格；写 Shader 材质做一些特效，比如发光、溶解、扫光这些视觉效果；还要负责性能优化，控制控件层级、用 Invalidation Box 这些机制减少不必要的重绘；有时候他们还会自己写工具或脚本，辅助批量导入 Figma 的资源。他们最后交出的是一套可以复用的 Widget 组件库，以及优化过的 UI 材质，交给 UI Programmer 去接上真正的逻辑。

### Slide 14｜UI Programmer
第五个角色，UI Programmer，也就是专门做界面实现的程序。他们在 UMG 里把前面做好的 Widget 组件拼装成完整的界面，处理好不同分辨率下的自适应；写蓝图逻辑处理点击事件、界面跳转、状态切换这些交互；还要处理手柄导航焦点、以及触屏、鼠标、手柄之间输入方式的切换。他们做出来的是一个能运行的 Widget Blueprint，但这时候界面上显示的数据可能还是假的，要等下一步真正的数据接进来。

### Slide 15｜Gameplay Programmer
第六个角色，Gameplay 或者 Systems Programmer，也就是负责游戏系统逻辑的程序。他们提供界面真正需要显示的数据，比如血量、背包物品、任务进度、货币数量，然后和 UI Programmer 一起把这些数据"绑定"到界面上，让界面能实时刷新。简单说，前面几个角色做出来的是一个漂亮但是空的壳子，Gameplay Programmer 就是负责把真实的游戏内容装进这个壳子里，让界面真正"活"起来。

### Slide 16｜支持团队
除了这六个主线角色，还有三个贯穿全程的支持角色。Localization 负责把界面文本翻译成多语言，还要检查不同语言下文字会不会溢出、换行错乱，甚至像阿拉伯语这种从右到左书写的语言排版问题，他们也会提前跟设计师沟通，让设计预留足够的文字扩展空间。QA 负责在不同分辨率、不同平台、不同设备上做实际测试，包括安全区、手柄导航、无障碍功能这些，还要做回归测试，防止一个改动引入新的问题。Producer 或者项目经理，则是全程排期、协调各个角色之间的交接时间点，跟踪工单进度，组织跨部门的评审和决策。

### Slide 17｜完整流程图
把前面讲的所有角色放在一张图上，大概是这样：Game Designer 出需求文档，交给 UI/UX Designer 做出 Figma 设计稿，UI Artist 做出最终美术资产，Technical Artist 转成 UMG 组件和材质；然后 UI Programmer 接上界面逻辑，同时 Gameplay Programmer 提供真实数据，两边互相配合；Localization 处理多语言文本，QA 全程给反馈。但这里要特别强调一点，这张图看起来是一条直线，但实际工作中完全不是。QA 发现的问题经常会打回给 UI Programmer，甚至打回给最早的 UX Designer 重新设计；Producer 在每一个交接点都要做协调。整个过程是不断迭代、不断往回修的，不是一次性从头做到尾。

### Slide 18｜团队规模如何影响分工
刚才讲的八个角色，是一个相对理想化、分工清晰的模型。但实际上，团队规模不同，这八个角色会被压缩或者拆得更细。举个例子，五到十五人左右的独立团队，往往是一个人身兼 UI/UX Designer 和 UI Artist，甚至连一部分 UMG 实现都自己做；程序这边可能就一个人，同时负责 UI 逻辑和游戏系统逻辑。这种团队几乎不会有专职的技术美术，一人多岗，交接成本几乎是零，但个人的工作量会非常大。到了三十到八十人左右的中型团队，就开始出现专职分工，可能有一到两个 UI/UX Designer，一个专职 UI Artist，一个技术美术（这个人常常也兼一部分 UI Programmer 的活），再加上两三个 Gameplay Programmer 分给多个系统共用。而三百人以上的大型 3A 团队，光是 UI 团队本身可能就有十到二十人以上，会细分成 UX Research、UI Design、UI Art、UI Engineering、Technical Art 这些小组，每个小组都有自己的 Lead，甚至还会有专职只做手柄导航测试，或者专职只做本地化排版的人。这里要提醒一下，这些人数和架构都是行业里常见模式的示意举例，帮助大家理解规模效应，不是某个具体公司真实的组织架构，实际情况会因为项目类型、发行阶段这些因素差异很大。

### Slide 19｜UI/UX 面试考什么
接下来聊聊找工作相关的内容。UI/UX 设计师岗位面试，一般会从这几个维度考察。第一是作品集深挖，面试官会让你详细讲一个项目从需求到最终落地的完整决策过程，尤其是玩家测试之后你做了哪些调整，为什么。第二是现场设计挑战，给你有限的时间设计一个具体界面，比如商店或者背包界面，然后当场讲你的思路和取舍。第三是工具熟练度，比如 Figma 里组件怎么用、Auto Layout 怎么搭、大团队的文件怎么组织管理。第四是看你能不能把 UX 的基本原理，比如 Fitts's Law、Hick's Law，用游戏场景讲清楚，而不是背概念。第五是平台适配的思维，同一个界面在手柄、触屏、鼠标下应该怎么调整。最后是协作和沟通能力，比如你怎么应对跟程序或者制作人的意见分歧。

### Slide 20｜UI/UX 常见真题
给大家举几个具体的真题例子。设计挑战类的，比如"给你 30 分钟，为一款手游设计一个抽卡或者商店界面，说说你的取舍"。批判分析类的，会直接给你看某个游戏的 HUD 截图，问你会改哪里、为什么。原理应用类的，会让你举一个自己用过的 UX 原理，说说它怎么影响了你的设计决定。无障碍相关的，会问你怎么为色盲玩家或者单手操作的玩家设计界面。还有行为面试类的问题，比如说一次你和工程师或者制作人在方案上有分歧，最后是怎么解决的。这些问题背后考察的其实都是你的判断力和沟通能力，而不只是审美。

### Slide 21｜技术美术要学什么
接下来讲技术美术这个角色，因为这是很多同学容易忽略、但其实是设计和工程之间非常关键的一环。他们需要学的东西大概分六块。编程基础，包括 Python 用来写工具和管线脚本，一些 C++ 或 C# 基础，以及在 Unreal 里深度使用 Blueprint。引擎的 UI 系统，比如 UMG 里的 Anchor 锚点系统、Invalidation Box、Retainer Box 这些机制，或者 Unity 的 UI Toolkit。Shader 和材质相关的知识，用来做发光、溶解、扫光这些 UI 特效。性能优化，包括 Draw Call 合批、Overdraw 分析，用 Unreal Insights 或者 Profiler 这些工具去排查界面卡顿的原因。工具开发能力，能自己写编辑器工具或者脚本，去自动化一些重复劳动，比如批量导入 Figma 资源。最后是跨部门沟通能力，因为他们经常要把设计师的想法翻译成技术上能实现的东西，也要把技术上的限制解释给设计师听。

### Slide 22｜技术美术面试怎么考
技术美术的面试会更偏技术和实操。常见的题目类型包括，让你介绍一个自己写过的管线工具，说清楚它解决了什么具体问题。性能排查类的题，比如"一个界面导致游戏掉帧，你会怎么一步步去定位原因"。概念理解类的，比如让你解释 Invalidation Box 或者 Retainer Box 的作用和使用场景。还会有现场编码环节，比如写一段脚本去解析 Figma 导出的 JSON 数据，生成对应的 Widget 结构骨架。也可能考系统设计能力，比如设计一个可复用的九宫格按钮组件，让设计师和程序都能直接拿去用，不需要每次都手动调蓝图。

### Slide 23｜笔试题一般考什么
除了面试聊天，很多公司在面试前还会加一轮笔试，考的内容因岗位而不同。UI/UX 设计师的笔试通常包括设计理论基础，比如色彩、字体排版、格式塔原理、栅格系统这些；游戏行业和竞品认知，看你对同类型产品的 UI 趋势熟不熟；案例分析或者说找茬题，给你一张界面截图，让你指出可用性和层级上的问题；现场排版或者线框题，给一批素材让你当场搭一版界面；交互流程图题，画出某个功能的用户流程；还有文案题，比如写一段错误提示或者引导语。技术美术的笔试就完全是另一个方向了，会考编程基础，Python 或者 C++ 的语法和简单逻辑题；数学基础，向量、矩阵、三角函数这些跟图形学相关的内容；Shader 或材质的概念题，让你解释某个视觉效果大概怎么实现；引擎机制问答，比如 UMG 或者 uGUI 底层是怎么运作的；还有找问题题，给一段有性能隐患的代码或者蓝图，让你指出瓶颈在哪。

### Slide 24｜校招 vs 社招：对应 Junior / Senior Artist
最后聊聊校招和社招的区别，这个在国外通常对应 Junior/Associate Artist 和 Senior Artist 这两个级别。校招面向的是应届生或者毕业一两年内的人，考察重点是基础功底扎不扎实、学习能力和可塑性强不强，作品集更看重的是思考过程，而不是有没有实际项目经验；通常会配一个 Mentor 带教一段时间，薪资相对较低，但晋升路径比较清楚；硬性要求上不需要你独立带过完整项目。社招对应的是 Senior，甚至再往上 Lead 或者 Principal 级别，通常要求两年以上经验，到 Senior 一般要三到五年以上；这时候考察重点变成了有没有真正上线过的项目经验、能不能独立负责一个完整模块、对整个管线——也就是从设计到实现——的理解深不深，还有能不能带新人或者推动跨部门协作。海外的 Senior UI Artist 通常还会要求你不只是画得好看，还要理解引擎实现上的限制，这也是为什么前面提到的"管线适配性"，在社招阶段会被看得格外重——校招看的是潜力，社招看的是你能不能马上独立干活。

### Slide 25｜申请欧美/英国设计院校：UI/UX 作品集要求
除了讲毕业以后找工作，也顺便讲讲很多同学接下来可能会面对的另一件事——申请欧美或者英国的设计院校，无论是本科转学还是读研，作品集要求跟公司招聘完全是另一套逻辑。提交形式上，通常要求 8 到 20 个项目，用 PDF 或者网站链接提交，可交互的原型可以直接发一个可点击的链接，很多学校其实并不强制要求你有代码基础或者特定软件背景。考核重点几乎所有学校都反复强调同一件事：草图、线框图、迭代过程比最终图好不好看更重要，很多学校还要求每个作品配一段反思短文，讲清楚你的设计决策是怎么来的。那 UI/UX 具体要做到什么程度呢？不需要到工作室那种能落地到引擎的水平，而是要能画出信息架构和用户流程图，能做出一版中低保真的可点击原型，有基础的排版和层级意识，不要求美术功底特别拔尖。最后说说为什么 Figma 在申请阶段也很重要：它能让招生官直接看到你从一个方案改到另一个方案的迭代过程；不用写代码就能做出可交互原型直接发链接体验；而且它本身就是入学之后大家默认使用的协作工具，提前熟悉了就能把精力放在设计本身上。

### Slide 26｜游戏设计 vs 游戏交互（UX）专业：作品集差异
那具体到游戏相关的专业，游戏设计专业和游戏交互，也就是游戏 UX 设计专业，两者的作品集要求其实差别挺大的。游戏设计专业，比如 Vancouver Film School、Abertay University、Breda University 这些学校，更看重的是游戏概念和设计能力本身，不强调视觉打磨。常见的提交形式是游戏概念提案，包括题材说明、玩法机制图、关卡和角色大纲、世界地图，或者是一份代码作品集，附上源码片段和文字说明。VFS 甚至允许用他们的 Foundation 先修项目来替代作品集要求。竞争比较激烈的项目，比如 Breda，在作品集之外还会加一轮面试，跟你讨论作品集里的创作过程。而游戏交互，也就是游戏 UX 设计方向，作品集要求就更接近我们前面讲的通用 UI/UX 标准了：线框图、可点击原型、用户研究记录，核心考核的还是设计思维过程，而不是画面好不好看。但游戏 UX 方向会额外要求你展示对游戏专属交互场景的理解，比如 HUD、菜单系统、新手引导、手柄导航这些，这是通用 UI/UX 作品集里不会特别强调的部分。Figma 或者类似的原型工具在这个方向上几乎是标配，能直接让人体验你设计的可玩界面流程。这里也提醒一下，具体学校的要求每年可能会有调整，实际申请前还是要以各校官网当年公布的为准。

### Slide 27｜给 VFS 同学的建议
最后给大家几条实际建议。第一，作品集要体现的是"决策过程"，不只是好看的图，面试官更想看到你为什么这么设计、中间改过几版、玩家测试反馈是什么，这些思考过程比最终图好不好看更重要。第二，建议大家至少熟悉一个引擎的 UI 系统，不管是 UMG 还是 Unity 的 UI Toolkit，哪怕做得不完美，这也能证明你懂"设计要怎么真正落地"，这是很多刚毕业的同学容易缺的一块。第三，了解一下你上下游角色具体在做什么，就算你只想做 UX 设计师，懂一点技术美术的限制，也会让你跟技术团队的沟通顺畅很多。第四，建议大家在校内项目里就练习模拟真实工作室的分工方式，哪怕是几个人小组一起做，也试着按设计、美术、程序这样的角色去分工交接，提前适应这种协作节奏。第五点，也是我想特别强调的一点：在真正的公司里，这八个角色是由好几个不同的人分工完成的，没有人会真的一个人全包。但是找工作的时候情况不太一样——面试官往往更青睐"一专多能"、也就是所谓 T 型的候选人，你的主科可能是 UI/UX 设计，但如果你还能展现出对整个管线，从 Figma 设计到 UMG 落地这个过程的理解，会让你比只会画图的候选人更有竞争力，尤其是中小型工作室，这种"管线适配性"经常是筛简历时的加分项。
