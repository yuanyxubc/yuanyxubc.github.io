### 【P4 · 00:11–00:15 · Project Brief / 项目 Brief：史莱姆人格研究所】

> **【📄 打开文档】** （可以现场打开团队自己提交的项目文档截图，或者直接用这一页的文字过一遍）

这是你们自己交上来的项目版本，比我们最初设想的要具体得多。

玩家扮演一只史莱姆，加入"怪物大学"，经历入学到毕业的四个阶段——春、夏、秋、冬，通过问答互动，最终揭晓自己的 MBTI 人格类型，还有对应的职业身份。平台是 Steam，桌面宠物这种形式，不是全屏游戏。美术风格是像素风，基础单位定在 32px——这个数字你们自己已经定了，后面讲 Figma 规范的时候会直接用上，不是我今天现造一个数字出来。

好，先看两个具体的机制，一个是核心玩法循环，一个是测试本身。

---

### 【P5 · 00:15–00:18 · Core Loop / 核心玩法循环】

这条循环走一遍：生成桌宠 → 史莱姆在桌面上移动 → 玩家点击 → 弹出问题 → 选择答案 → 播放升级特效 → 展示偏移面板 → 这样循环推进四季 → 毕业 → 揭晓 MBTI 和对应职业 → 之后史莱姆常驻桌面，点击还能看回忆。

我带大家念一遍这条循环，主要是想确认一件事：全班对这个项目机制的理解是同步的。接下来我们要用理论去拆解它，如果连机制本身都没对齐，拆解就会拆错方向。

---

### 【P6 · 00:18–00:20 · The Test / 16 题加权偏移测试】

测试本身也值得单独说一下，因为它比一般的问答测试要复杂一点。

一共 16 道题，每道题对应 MBTI 的一个维度。关键是——不同选项造成的偏移程度是不一样的，不是简单的二选一、非黑即白，是加权计分。而且有些题目不止影响测试结果，还会影响场景画面本身，比如某道题选了"捡到小花"，场景里就真的会多一朵花出来。

这条机制我留到后面讲"结构层"和"表现层"的时候会再提一次——它正好能说明这两层是怎么互相影响的，先记住这个例子。

---

### 【P7 · 00:20–00:24 · Spot The Framework / 找找看：你们已经在用哪些框架了】

在正式进入五要素拆解之前，我想先做一个小实验。

四季结构、偏移面板、场景累积装饰、毕业后回忆——这些细节，其实都是你们自己在不知不觉中，做了 Structure、Scope、Emotion 层的设计决定，只是当时没有意识到自己在用这套框架。

> **【✋ 互动】** 两人一组，从项目 Brief 里挑一个细节，说出它对应五要素里的哪一层。给 3 分钟，抽 1-2 组说结果。
> 兜底：如果学生说不准也没关系，重点是让他们意识到"我已经在用这套东西了"，答案精不精确没那么要紧。

这个环节的意义在于建立信心——给已经在做的事情，找到一个更专业的说法，你们其实已经在用这套框架了，只是缺一个名字。接下来我们就正式一层一层拆。

### 【P8 · 00:24–00:27 · Five Elements: Strategy / 策略层】

先看策略层，这是整个拆解的地基。

这不是一款需要专心坐下来玩的游戏，它更像一个陪伴型、低压力的桌面常驻应用，套了一层"怪物大学"的轻松世界观。它的承诺是——轻量的自我探索，加上陪伴感，再加一点点养成的仪式感，不是硬核挑战。

目标受众也偏泛：桌宠爱好者，加上对 MBTI、人格测试感兴趣的人。这两点会决定后面所有设计的基调，所以先把它记住。

---

### 【P9 · 00:27–00:29 · Five Elements: Scope / 范围层】

范围层回答的是：这个项目需要哪些系统。我们数一下：史莱姆状态系统、问答系统（16题加权）、成长/阶段系统（四季）、事件分支系统、场景装饰累积系统、右键菜单、MBTI 偏移面板、结果揭晓页、毕业后回忆对话、存档、还有分享出口。

我特意把这十一个系统都念出来，是想让大家意识到一件事：这不是"画几张图"的工作量，是一整套系统设计。今天和接下来几节课要做的，就是把这十一个系统一步步落地。

---

### 【P10 · 00:29–00:32 · Five Elements: Structure / 结构层】

结构层看的是流程：首次启动引导 → 桌宠常驻、不定时弹出问题气泡 → 回答，分布在四季推进中 → 每次小成长反馈 → 循环推进，直到四季走完 → 毕业典礼 → 结果揭晓页 → 之后常驻在"已毕业"状态，点击可以看回忆。

这条流程建议现场画一条时间线，比纯听文字更直观——你们可以拿这条线去对照自己项目的实际流程图，看看有没有漏掉的节点。

---

### 【P11 · 00:32–00:34 · Five Elements: Skeleton / 框架层】

框架层这里有个这个项目特有的地方。因为是桌面常驻应用，屏幕上大部分时间只有史莱姆本体，加上累积的场景小装饰，其它 UI——问题气泡、偏移面板、右键菜单、结果页——都是按需弹出，不常驻。

这可以呼应 Lesson01 讲过的一个点，"UI 越少，情绪有时越强"，我们当时举的例子是旷野之息。但这里的理由不太一样：不是刻意做减法追求氛围，是桌面场景本身就决定了不能有一个常驻的大界面——你不可能让史莱姆占着半个屏幕。

---

### 【P12 · 00:34–00:37 · Five Elements: Surface / 表现层】

表现层落到具体做法：像素风，32px 基础单位，史莱姆要有 Q 弹的挤压拉伸动效，这个可以呼应 Lesson01 讲过的"Game Juice"、打击感的概念。配色偏温暖治愈，毕业揭晓那一刻要做成一个仪式感时刻。

还有一点容易被忽略：场景累积的小装饰，也要保持统一的像素风格和色彩基调，不能各画各的。这条其实已经在预告下午要讲的 Design Tokens 了——先记一个印象，等下会正式展开。

### 【P14 · 00:42–00:45 · Game UX: What To Diagnose / 这个项目要重点诊断什么】

五要素讲完了，我们换一把尺子——Game UX，看看这个项目具体要诊断什么。

可用性这块，核心问题是：弹窗时机会不会打扰玩家正在做的事。桌宠类应用的标准交互是右键菜单，设置、退出这些功能都收在这里，这是一个平台惯例，不是你们自创的。

动机这块，是养成循环加上好奇心——我会测出什么类型、什么职业——再加一点收集心态，16 种人格天然带收集感。

情感这块，整体基调是治愈温暖，毕业要有仪式感，回忆对话是额外的情感设计点，让玩家可以持续回味这只史莱姆的成长过程。

心流这块要单独提醒一下：这不是 Lesson01 讲的战斗心流，是"日常陪伴的节奏感"——多久弹一次问题、每次占用玩家多长时间。这条和上节课讲的完全不是一回事，别直接套错模板。

## 第二节 · 视觉理论 + Figma 生产规范 + FigJam 主工作坊（00:55–02:28）

### 【P19 · 00:55–00:59 · Color Theory In Practice / 色彩理论落地】

欢迎回来。第二节我们讲两块理论——色彩和字体，然后大部分时间留给 Figma 生产规范和动手。

> **【📄 打开文档】** 从《课堂材料索引》进入：`FIGMA → 05 Figma 操作指南 → OPEN`。打开后找“设置 Design Tokens：颜色样式”部分。那份文件把选色、建 Color Styles、检查对比度写成了逐步操作。

先说方法：我们不用“我觉得这个颜色好看”来选色，而是用 HSB 模型来拆。H 是 Hue，色相，决定这个颜色大概属于红、黄、绿、蓝哪一类；S 是 Saturation，饱和度，决定它是鲜艳还是灰一点；B 是 Brightness，明度，决定它是亮还是暗。

这三个维度分开之后，选色就会可控很多。比如你想给 16 种人格做区分，不一定每一种都重新选一套完全不同的颜色；可以先用色相区分大类，再用饱和度和明度区分强弱。四季也是同理：春天可以偏暖、偏亮，冬天可以偏冷、偏低饱和，但它们仍然要像同一个游戏里的颜色。

这个项目的基础调性是温暖、治愈、轻量陪伴，所以主色建议先用类似色方案。类似色就是色相环上相邻的一组颜色，优点是统一、不刺激，适合桌宠这种常驻在屏幕上的东西。互补色和三角色不是不能用，但更适合作为少量强调：比如危险提醒、成功反馈、稀有人格标签。强调色如果铺满全屏，就不再是强调，会变成噪音。

接下来把颜色变成系统。最低限度要定 7 个语义角色：Primary、Background、Surface、Text、Accent、Danger、Success。这里的重点是“按用途命名”，不是按视觉印象命名。不要叫“粉色1”“蓝色2”，因为几周后你自己也不记得它们应该用在哪里。叫 Primary，就表示它是主操作和品牌识别；叫 Surface，就表示它是卡片或面板背景；叫 Danger，就表示它只服务警告或错误。

最后做两个检查。第一，文字对比度：正文和背景至少要达到 4.5:1，大号标题至少 3:1，这是 WCAG 对文本可读性的最低标准。第二，不要只靠颜色传达状态。比如“已选中”和“未选中”除了颜色不同，也可以有描边、图标、文字标签或位置变化。这样色弱学生也能看懂，游戏放到不同屏幕上也更稳定。

### 【P20 · 00:59–01:02 · Type & Layout / 字体与排版】

字体这块也一样，不是凭感觉每一页随手调。先定一套四级字号阶梯：Heading、Subheading、Body、Caption。你们可以先用一个稳定起点：Heading 24/28，Subheading 18/22，Body 14/20，Caption 12/16。前面的数字是字号，后面的数字是行高。后面如果字体本身偏大或偏小，再微调。

层级不要只靠“字大一点”。标题、正文、说明文字之间，通常是字号、字重、颜色深浅一起工作。比如标题可以更大、更粗、颜色更实；正文用稳定字重；说明文字可以小一点、颜色淡一点。这样界面就有阅读顺序，玩家先看到该看的东西。

像素字体有一个坑：它通常只在原生尺寸或整数倍下清晰。12px、24px、36px 这类整数倍通常更稳；13.5px、19px 这种非整数倍很容易糊。这个纪律和后面讲的 32px 尺寸阶梯是同一件事：像素风不是“看起来方块”，而是所有尺寸都尊重像素网格。

这个项目还有一个特殊场景：大量文字会出现在小对话气泡里。问答气泡、MBTI 偏移面板、毕业后回忆对话，都不是大段文章页面。你们不能只用“示例文字”测试排版，要放真实句子进去：短问题、长问题、两行回忆文本、人格解释。看它会不会溢出，会不会太挤，会不会一眼读不完。

所以字体这部分最后要交付的不是“我选了一个可爱字体”，而是一套规则：哪一级文字用在哪，字号是多少，行高是多少，像素字体用在哪些地方，正文是否需要换成更清楚的字体。这套规则之后要直接变成 Figma 的 Text Styles。

### 【P23 · 01:15–01:18 · Core Rules: Naming & Variants / 核心规则：命名 + 状态变体】

### 【P24 · 01:18–01:21 · The 32px Ladder / 32px 尺寸阶梯】

这一条是直接回应你们自己提出的问题——"怎么确认最小元素的大小，确保在不同屏幕上体验都方便"。

答案不复杂：你们已经确定美术基础单位是 32px，那全项目的间距、留白、图标尺寸，都应该是 32px 或者 16px 的倍数。具体的间距阶梯是 8、16、32、64px。可点击元素的最小点击区域，不小于 32px。

不用去解决"任意屏幕都完美适配"这种更复杂的问题，先定死一个目标分辨率，所有设计按这个来做就够了。操作上，在 Figma 里可以给目标 Frame 加 32px Grid；量间距时按住 Alt 或 Option 看像素距离。像素风素材导出时还要检查最近邻，也就是 Nearest neighbor，避免被默认平滑缩放磨糊。

这条阶梯，等下临摹的时候也要用上——量一量你临摹的案例，间距大概是不是这套倍数关系。

### 【P25 · 01:21–01:24 · Figma ⇄ UE / 控件标准对照】

这一条是这次新加的重点：让你们在 Figma 里画的东西，未来真的能被翻译成 UE 的 UMG 控件结构，不是画完才发现没法实现。

简单对照一下：自由定位的元素，对应 Canvas Panel；需要自动排列的内容，对应 Auto Layout 或者 Box；锁定尺寸不随内容变化的，对应 Size Box；需要跟内容拉伸的面板，比如对话框背景，要做成 9-Slice 素材，四个角不变形、边和中间可以拉伸。

这里再补两个操作点。第一，Constraints 在 Figma 里对应 UE 里的 Anchor：按钮是贴左上角、居中，还是跟右边界保持距离，要从现在就标清楚。第二，Auto Layout 只加在会随文字变长而伸缩的东西上，比如对话气泡和按钮组；史莱姆本体这种像素精灵图不要加 Auto Layout，否则网格会乱。

这一块的完整对照表和标注清单，同样在母版文件里，等下有空可以照着标一遍。

### 【P29 · 01:42–01:57 · Step 2: One Game Screen Analysis / 第二步：做一页游戏 UI 分析】

现在第二步，选一个真实游戏界面截图，做一页非常小的 UI 分析。这个页面不需要长，结构固定就行：

第一，放一张截图。第二，在截图上标三个点。第三，每个点写一句分析。第四，最后总结一条可复用设计规则。

每一句分析都用同一个句式：我看到什么，它为什么有效或有问题，我能从里面学到什么规则。

举个例子：`我看到奖励弹窗先显示稀有度颜色，再显示数值；这让玩家先感受到奖励等级，再读具体信息；所以我的规则是：奖励反馈要先给情绪判断，再给数值细节。`

> **【✋ 互动】** 给 15 分钟。每个人做一页游戏 UI 分析，直接贴到 FigJam 自己的 `Analysis / 分析` 格子里。最少要求：1 张截图、3 个标注、1 条设计规则。做不完也要先贴出来，不要藏在电脑里。

### 【P31 · 02:04–02:08 · Case Criteria / 案例标准：什么样的案例可以临摹】

### 【P32 · 02:08–02:28 · Trace Task / Lesson02 小作业：临摹，或者直接原创】

两条路选一条：第一条，从你刚才圈出来的局部组件开始临摹；第二条，不临摹也可以，直接在自己负责的组件上做原创设计。两条路都算数，选你觉得对自己更有帮助的那条，不是必须临摹。

> **【✋ 互动】** 个人在自己的 Figma 里开一个新文件——注意是你自己的文件，不是团队共用的那个。给 20 分钟，用刚讲的 32px 阶梯和 Variants 思维，做你选的那条路。时间到不要求 100% 完成，重点是这个动作本身在训练你对间距、颜色和状态的手感。做完把你这个文件的分享链接贴到 FigJam 第二周图版自己的 Figma 格子里。

如果有人还没开过 Figma 文件，就按 `FIGMA → 05 Figma 操作指南 → OPEN` 的前几步走：新建 Design file，建 Design Tokens / Components / Wireframes / Hi-Fi / Prototype / Handoff 这些 Page，再建一个目标 Frame。今天这 20 分钟不要求你把整个项目文件都搭完，但至少要有一个可打开的个人 Figma 文件，一个清楚命名的 Frame，以及一个正在做的组件。

提前说清楚：不要求做完，也不要求临摹得多像。学生容易卡在"临摹得不像"或者"不知道原创画什么"，你要反复提醒——这一步看的是你有没有开始建立自己的 Figma 证据，不是最后那张图完不完美。后面真正打分时，会看你能不能讲清楚：你临摹学到了什么手法，或者你原创解决了什么具体问题。

再强调一次这是个人独立完成、独立提交的作业，不是团队共用文件里的一部分——链接要是你自己那份文件的。

第二节到这里结束，短休 5 分钟，回来我们继续这份小作业，然后收尾今天的安排。

> ## 附：给老师的几个提示
> 1. 这次的核心是"设计自己的项目"，而分析别人的案例只是练手的起点，学生的挫败点会不一样——更多是"不知道从哪下手"，比"看不懂案例"更常见。多用"先填表、再动手"这个节奏帮他们拆解模糊感。
> 2. 桌宠这个形式对大多数学生是新鲜的，建议课前自己先找 2-3 款真实桌宠类产品截图或录屏，上课时现场过一遍，帮学生建立"这个品类长什么样"的直觉。
> 3. 第二节的动手练习时间比较紧（20 分钟做 1 个局部组件），如果发现普遍进度跟不上，可以把"抽组展示"那部分挪到第三节开头，把第二节的动手时间往后借一点，不用卡得太死。
> 4. 学生自己报的问题——流程乱、没规范、不知道怎么定尺寸——是最好的教学切入点，直接引用他们自己的话来讲这一章，比泛泛而谈"行业规范很重要"更有说服力。
> 5. 建议提前把 `UIUX_02_材料索引.html` 发给学生。上课时不要报长文件名，直接报菜单路径：`FIGMA → 04`、`FIGMA → 05`、`TASK → 06`、`TASK → 07`。学生只需要点右侧 `OPEN`。
